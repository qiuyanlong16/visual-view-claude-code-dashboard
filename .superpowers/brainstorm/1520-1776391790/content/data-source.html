Recommended Data Source: Hook + Local WebSocket Server

Best balance of real-time, reliability, and implementation complexity

Why Option A (Hooks) is recommended

Pros

  • Native integration — Claude Code's hooks (settings.json) fire on session start, turn end, subagent start/end — the natural event boundaries you want to monitor
  • Real-time — hooks fire immediately, events pushed via WebSocket to the web UI
  • Low complexity — a small hook script sends JSON to a local server; no protocol design or MCP spec to follow
  • Memory access — hooks can read .claude/ directory contents (memory files, plans, etc.) and include them in events
  • No Claude Code config changes per project — hooks are set once globally

Cons

  • Granularity limited to hook trigger points — can't capture every single tool call mid-turn
  • Transcript data (full turn content) is only available at turn boundaries, not streaming

Architecture Overview

Claude Code ──(hooks fire)──► Hook Script
Hook Script ──(HTTP POST)──► Local Node Server
Local Node Server ──(WebSocket)──► Web UI Dashboard
Web UI ◄── real-time events ◄── Server
