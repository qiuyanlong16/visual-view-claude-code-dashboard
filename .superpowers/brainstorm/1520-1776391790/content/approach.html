Architecture Approaches

3 ways to structure the event server + hook integration

Option A: Hook Scripts → Node Express Server → Svelte SPA

Flow: Claude Code fires hooks → small Node.js scripts POST events → Express server receives events, persists to SQLite, serves SSE stream → Svelte app connects via EventSource

  • Pros: Clean separation, SQLite gives persistent history, mature ecosystem
  • Cons: Two processes (hook script + server), more moving parts
  • Best for: Projects where event history needs to survive server restarts
RECOMMENDED

Option B: Hook Script → Tiny Hono Server (edge-style) → Svelte SPA

Flow: Same hooks → POST to a Hono (Node runtime) server → in-memory event buffer + JSONL file for persistence → SSE endpoint → Svelte app with stores

  • Pros: Minimal footprint, Hono's SSE helper is elegant, JSONL is perfect for event streams (append-only, easy to replay), single dependency tree
  • Cons: JSONL has no query layer (fine for replay, harder for analytics without loading all events)
  • Best for: This project — dashboard + analytics where recent events matter most

Option C: All-in-One Svelte Dev Server with Built-in API

Flow: Hook scripts POST to the Vite dev server itself (using +server.js routes) → Svelte handles both UI and API in one process

  • Pros: Single process to start/stop, simplest dev experience, SvelteKit handles SSE natively
  • Cons: Dev server in production needs replacement, Vite dev server isn't meant for long-running servers
  • Best for: Rapid prototyping, not for a persistent monitoring tool

I recommend Option B — Hono gives you a lightweight server that can run long-term, JSONL is the right data structure for event streams, and it cleanly separates the API server from the Svelte frontend for deployment flexibility.
